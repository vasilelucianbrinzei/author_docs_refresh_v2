# Start Here

## Introduction

Use this section as the entry point to the LiveLabs author guide. It explains what the documentation covers across the end-to-end author workflow, what a production-ready outcome looks like, and how the guide is organized. Use it to understand where the GitHub and WMS background pages fit, when to move into the core workflow, and which later sections are supporting references rather than part of the standard author path.

Estimated Time: 10 minutes

### Objectives

* Understand what this documentation set covers
* Know the expected end result of the author workflow
* Know where to go for background, the core workflow, and supporting reference material

## Task 1: Understand What This Guide Covers

1. This guide supports LiveLabs workshop authors from initial WMS request through GitHub setup, content development, QA, publishing, and ongoing maintenance.
2. It is meant to be a practical working reference while you build and maintain workshops, not a document you read once and leave behind.
3. The standard author path lives in **Core Workshop Flow** and **Validation, QA, and Publish**. The GitHub and WMS pages in this opening section are short background references you can use when you need context before execution.

## Task 2: Know The Expected End Result

1. Your workshop should exist in the correct `oracle-livelabs` repository with a clean folder structure, a working `manifest.json`, and production-ready Markdown.
2. Your WMS entry should have the correct title, abstract, outline, prerequisites, tags, owners, and status.
3. Your workshop should pass self-QA, stakeholder review, and pull request checks before you request publishing.
4. After publishing, the same documentation set remains useful for maintenance, quarterly QA, and future updates.

## Task 3: How To Use This Guide

1. Start here to understand the scope, structure, and reading order of the documentation set.
2. Review the GitHub orientation page if you need repository, fork, clone, sync, or preview context before authoring.
3. Review the WMS orientation page if you need a quick understanding of statuses, roles, or where WMS actions fit in the lifecycle.
4. Move into **Core Workshop Flow** for the standard author path.
5. Use **Workshop Components and Reuse**, **Tools**, **Specialized Workflows and Reference**, and **Help, FAQ, and Support** only when those topics become relevant.
