Layer 1 / WMS gate

Submit Workshop Request

Create the request in WMS with enough clarity that reviewers know the learner outcome, owner, and delivery shape before you start building.

Output: approved request Time: 5 to 10 min Unlocks: repo work
Shortcut bubbles
The goal at the first layer is not to write the whole workshop. It is to create a clean, reviewable request with the right owner, outcome, and notes on anything unusual such as embedded HTML or media.

What to do

  1. Open WMS and create the workshop request with a title that says the exact learner outcome.
  2. Write a compact abstract that explains who the workshop is for, what gets built, and what the reader leaves with.
  3. Add the correct owner, reviewer, business contact, and any timing notes so approval does not stall on missing names.
  4. Call out nonstandard elements early, such as embedded HTML, autoplay video, FreeSQL, or special media needs.
  5. Submit the request and keep WMS as the system of record before heavy GitHub work begins.
Copy-ready starter brief

WMS request skeleton

Workshop Title:
Autonomous Database Quickstart for New Authors

Abstract:
A short workshop that shows a first-time author how to request, build, review, and publish a LiveLabs workshop.

Notes:
- Guided and reference author paths
- GitHub Pages prototype with scoped HTML, CSS, and JS
- Any embedded media called out before review

Quick checklist

  • Title is specific and readable to reviewers.
  • Abstract names the audience, outcome, and scope.
  • Real owner and contact fields are set.
  • Special HTML or media needs are visible in the request.
  • Status is submitted or approved before build work starts.

Common mistakes

  • Submitting a vague request with no clear learner outcome.
  • Starting the repo build before the workflow record is aligned.
  • Waiting until final review to mention special media or embedded HTML.

What good looks like

A reviewer can understand in under a minute what the workshop teaches, who owns it, and whether anything in the build will need special handling later.

This layer exists so the rest of the work starts from an aligned request instead of assumptions.

Layer 2 / GitHub build

Create Workshop + Add to GitHub

Set up the authoring environment, copy the sample structure, wire the manifest, and get a preview running early while the workshop is still cheap to fix.

Output: live preview Time: 15 to 20 min Unlocks: content authoring
Shortcut bubbles
Most new-author friction lives here. The trick is not to perfect everything at once. Build the skeleton fast, confirm preview works, then deepen the content only after the structure is stable.

What to do

  1. Create or fork the correct repository and open it in GitHub Desktop or VS Code.
  2. Copy the sample workshop structure into a lowercase folder path and rename it cleanly.
  3. Update manifest.json with the real workshop title, help address, and tutorial order.
  4. Wire the 01-start-here page and first lab files before deep writing so preview fails early, not late.
  5. Commit, push, and validate the GitHub Pages preview as soon as the skeleton exists.
Starter manifest

Minimum structure for a first build

{
  "workshoptitle": "My Workshop Title",
  "help": "my-team@oracle.com",
  "tutorials": [
    {
      "title": "Start Here",
      "filename": "./../../01-start-here/start-here.md"
    },
    {
      "title": "Core Workflow",
      "filename": "./../../02-core-workflow/core-workflow.md"
    }
  ]
}

Quick checklist

  • Repository path and branch are correct.
  • Folder and file names stay lowercase and case-safe.
  • Manifest order matches real files.
  • Preview loads before long authoring sessions begin.
  • Images and assets already have predictable folders.

Common mistakes

  • Using the wrong repository path or missing a case-sensitive rename.
  • Authoring for too long before checking the first real preview.
  • Leaving sample manifest fields or broken file paths behind.

What good looks like

You can open the workshop preview, the manifest points to real labs, and the structure is clean enough that content work can continue without foundational rework.

This layer combines environment setup, structure, manifest wiring, and early preview into one build checkpoint.

Layer 3 / QA chamber

Review & Publish

Run preview and QA checks, complete Self QA in WMS, open a clean PR, and supply the final publishing details so the workshop can move to production.

Output: reviewed release Time: 10 to 15 min Unlocks: production
Shortcut bubbles
The deepest layer is not more content. It is more precision. Reviewers should see a working preview, a clean workflow state, a safe PR, and the production URL pattern they expect.

What to do

  1. Run the preview end to end and append ?qa=true to open LintChecker while fixes are still cheap.
  2. Complete the Self QA checklist in WMS, save it, and only then mark Self QA Complete.
  3. Open a clean pull request with the WMS ID, reviewer context, and a stable preview URL.
  4. Update the Publishing tab with the final production URL and any required release notes.
  5. After merge, verify the production workshop and leave WMS ready for later quarterly QA.
Review handoff

PR and preview pattern

PR title:
Publish My Workshop Name (WMS 12345)

Preview URL:
https://<github-username>.github.io/<repo-name>/<path>/workshops/<variant>/index.html?qa=true

Production URL:
https://oracle-livelabs.github.io/<repo-name>/<path>/workshops/<variant>/

Quick checklist

  • Preview works end to end and LintChecker issues are resolved.
  • Self QA checklist is saved before any status change.
  • PR title includes the WMS ID where reviewers expect it.
  • Publishing tab matches the final production URL.
  • Production workshop is verified after merge.

Common mistakes

  • Sharing a preview before checking tutorial order, links, images, and copy flows end to end.
  • Changing WMS status before the checklist is saved and certified.
  • Mixing development and production URLs or opening a PR from a stale branch.

What good looks like

Reviewers can approve from one clean PR, the WMS record is consistent with the release, and the production workshop matches the preview you already checked.

At the end of this layer, the workshop is ready either to publish or to branch into the reference toolkit for a targeted feature lookup.
