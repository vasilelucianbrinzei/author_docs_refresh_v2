Layer 1 / WMS gate

Submit Workshop Request

Create the request in WMS with enough clarity that reviewers know the learner outcome, owner, and delivery shape before you start building.

Output: approved request Time: 5 to 10 min Unlocks: repo work
Shortcut bubbles
The goal at the first layer is not to write the whole workshop. It is to create the real WMS record that reviewers, stakeholders, and the later QA flow will all use.

What to do

  1. Connect to Corporate VPN, open Oracle Workshop Management System, and click Submit a New Workshop Request.
  2. Fill Workshop Basic Information completely, especially Stakeholder, Workshop Council, Workshop Owner Group, Workshop Abstract, Workshop Outline, and Workshop Prerequisites.
  3. Open the Tags tab and set the required Level, Role, Focus Area, and Product tags before you click Create.
  4. Use the Go to Market - Social tab to add blog and promotion details before publish time if the workshop needs them.
  5. Wait for the request to move through Submitted and Approved before you treat GitHub work as the main source of truth.

Quick checklist

  • The title names the learner outcome, not just the product.
  • The abstract, outline, and prerequisites are filled thoroughly.
  • Stakeholder, council, and owner group are correct before review starts.
  • Required tags are set.
  • The request is visible in Submitted or Approved before heavy build work starts.

Common mistakes

  • Writing a vague abstract and forcing the council to ask for more information.
  • Starting repo work before the WMS record is aligned with the real workshop idea.
  • Forgetting tags, social details, or special delivery notes until the end of the flow.

What good looks like

A council reviewer can tell in under a minute what the workshop teaches, who owns it, and whether anything about the later build or publish path needs special handling.

This layer exists so the rest of the work starts from an aligned request instead of assumptions.

Layer 2 / GitHub build

Create Workshop + Add to GitHub

Set up the authoring environment, copy the sample structure, wire the manifest, and get a preview running early while the workshop is still cheap to fix.

Output: live preview Time: 15 to 20 min Unlocks: content authoring
Shortcut bubbles
Most new-author friction lives here. Build the workstation, repo, structure, and preview first. Deep writing comes after the skeleton proves it can render.

What to do

  1. Create or confirm the GitHub account that uses your Oracle email, install GitHub Desktop, and sign in from File > Options > Sign in.
  2. Install Visual Studio Code plus Live Server, then set Markdown indentation to tabs size 4 before you create nested steps or code blocks.
  3. Fork the correct oracle-livelabs repository, clone your fork, and run git config --global core.longpaths true plus git config --global core.ignorecase false when you need them.
  4. Copy the sample workshop structure, keep all files lowercase, and wire manifest.json to real section or lab files instead of placeholders.
  5. Enable GitHub Pages on your fork and validate the preview URL before you spend hours writing content.

Quick checklist

  • GitHub Desktop, VS Code, and Live Server are working.
  • The fork, local clone, and preview URL all point to the same workshop path.
  • Folder and file names stay lowercase and case-safe.
  • Manifest order matches real files.
  • Preview loads before long authoring sessions begin.

Common mistakes

  • Using the wrong GitHub account or the wrong repository path.
  • Skipping the daily merge and previewing only after deep edits.
  • Leaving sample manifest fields or broken file paths behind.

What good looks like

You can open the preview from your fork, the manifest points to real labs, and the structure is stable enough that deeper content work will not require foundational rework.

This layer combines environment setup, structure, manifest wiring, and early preview into one build checkpoint.

Layer 3 / QA chamber

Review & Publish

Run preview and QA checks, complete Self QA in WMS, open a clean PR, and supply the final publishing details so the workshop can move to production.

Output: reviewed release Time: 10 to 15 min Unlocks: production
Shortcut bubbles
The deepest layer is not more content. It is precision. Reviewers should see a stable preview, a saved checklist, a clean PR, and the exact production URL pattern they expect.

What to do

  1. Run the preview end to end, append ?qa=true, and fix LintChecker findings before you ask anyone else to review the workshop.
  2. On the WMS details page, update title, description, outline, prerequisites, tags, and development URL before you open the Self QA Checklist.
  3. Complete every checklist checkbox, add the PR link plus your personal github.io workshop link, upload the requested evidence, and click Save.
  4. Only after the checklist save succeeds should you move to Self QA Complete, create the PR with the WMS ID in the title, and request publishing from the Publishing tab.
  5. After merge and publish approval, verify the oracle-livelabs production workshop and leave WMS aligned for later quarterly QA.
UI you should see

Checklist, PR, and publishing evidence

Review handoff

PR and preview pattern

PR title:
Publish My Workshop Name (WMS 12345)

Preview URL:
https://<github-username>.github.io/<repo-name>/<path>/workshops/<variant>/index.html?qa=true

Production URL:
https://oracle-livelabs.github.io/<repo-name>/<path>/workshops/<variant>/

Quick checklist

  • Preview works end to end and LintChecker issues are resolved.
  • Self QA checklist is saved before any status change.
  • PR title includes the WMS ID where reviewers expect it.
  • Publishing tab matches the final production URL.
  • Branch is current and conflict free before approval is requested.
  • Production workshop is verified after merge.

Common mistakes

  • Sharing a preview before checking tutorial order, links, images, and copy flows end to end.
  • Changing WMS status before the checklist is saved and certified.
  • Mixing development and production URLs or opening a PR from a stale branch.

What good looks like

Reviewers can approve from one clean PR, the WMS record is consistent with the release, and the production workshop matches the preview you already checked.

At the end of this layer, the workshop is ready either to publish or to branch into the reference toolkit for a targeted feature lookup.
