# Specialized Workflows

## Introduction

Use this section for advanced or niche workflows that only apply to some workshops. These topics are intentionally separated from the core author path so most authors can stay focused on the standard build-review-publish route unless the delivery model really changes.

Estimated Time: 10 minutes

## Objectives

* Identify which advanced workflows are actually relevant to your workshop
* Keep specialized setup separate from the standard author path
* Find the right reference quickly when your delivery model requires it

## Task 1: Decide Which Delivery Model Actually Applies

| Delivery model | Start with | Use it when |
| --- | --- | --- |
| Sprint instead of workshop | Lab 16 | The deliverable is short-form sprint content, not a standard workshop |
| Browser-based graphical desktop | Lab 17 | Learners need a preloaded desktop or browser-accessible UI |
| Custom image and Marketplace path | Labs 18-20 | The workshop depends on a custom image and Marketplace publication |
| Secure Desktop access | Labs 21-22 | Participants are blocked by restricted corporate devices or network policy |

## Task 3: Keep Specialized Work Separate From The Main Flow

1. Complete the core author path first unless the specialized workflow changes your structure from day one.
2. Use the specialized labs that match your workshop delivery model and skip the rest.
3. Return to **Validation and Publish** once the specialized setup is stable enough for review.

## Task 4: Before You Leave Specialized Workflows

Use this checkpoint before you move back into the main release flow:

* The specialized path is actually required by the workshop delivery model.
* Only the relevant labs were opened and the unrelated branches were skipped.
* Preview, URLs, or environment behavior were tested before the publish request stage.
* WMS, publishing, and support expectations still match the delivery model you chose.
