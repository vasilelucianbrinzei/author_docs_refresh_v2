# Core Workflow

## Introduction

This section is the hands-on author path. Use it for the daily authoring loop: request the workshop, prepare the workstation, stay in sync with GitHub, and build markdown in a previewable structure before you move into QA and publish.

Estimated Time: 10 minutes

## Objectives

* Know the required author path for most workshops
* Know what each core lab covers
* Know when to pull in supporting sections without losing the main flow

## Task 1: Use This Section For The Main Authoring Loop

1. Start here after **Start Guide** once you understand the WMS lifecycle and GitHub model.
2. Stay in this section while the workshop is still being requested, structured, written, and previewed.
3. Do not move to **Validation and Publish** until the preview already works and the workshop structure is stable enough to test end to end.

## Task 3: Pull In Supporting Sections At The Right Time

| If you need... | Go to... | Why |
| --- | --- | --- |
| Review, self-QA, publishing, or timeline expectations | Validation and Publish | It covers QA, publishing, and the timeline details page |
| Optional interactive or reusable workshop elements | Reuse and Enhancements | It covers FreeSQL and quizzes |
| Better screenshots or smaller image files | Tools and Productivity | It covers capture standards, optimization, and cleanup |

## Task 4: Before You Move On To Validation

Use this checkpoint before you leave the core workflow:

* The WMS request has real title, outline, prerequisites, owner fields, and tags.
* GitHub Desktop, VS Code, and Live Server are already installed and working.
* The repo is synced with upstream and the GitHub Pages preview opens correctly.
* The workshop uses the sample structure cleanly and the manifest points to the real labs.
* Preview with `?qa=true` already surfaces the right workshop order, help email, and validator hints.

## Task 5: Common Misses In This Section

* Starting deep markdown work before the workshop request or repo setup is stable.
* Copying an older workshop instead of the current sample structure.
* Waiting too long to test GitHub Pages preview and discovering path or case issues late.
