# Review and Publishing Timelines

## Introduction

Use this page when you need the expected turnaround time for common LiveLabs handoffs such as request review, PR review, stakeholder QA verification, and publishing. This reference belongs with the validation and publish workflow. Treat these as planning expectations, not guarantees.

Estimated Time: 5 minutes

### Objectives

* Review the main LiveLabs service-level expectations
* Know when a follow-up is reasonable
* Avoid escalating work before the expected window has passed

## Task 1: Review The Main SLAs

| Process | SLA | Notes |
| --- | --- | --- |
| GitHub PR review | 1 business day | LiveLabs admin team reviews incoming PRs. |
| Workshop submission review | 2-3 business days | Council approves or requests more information. |
| Stakeholder QA verification | 2 business days | Stakeholders verify the Self QA submission. |
| Move to `Completed` | 1 business day | After stakeholder QA verification is complete. |
| Workshop publishing | 1 business day | After the publishing request is approved and the workshop is ready for production. |
| Event code review | 1 business day | LiveLabs reviews the request before enablement. |
| WMS to production sync | Within 1 day | Metadata changes may take time to appear in production. |
| Email support response | 1-2 business days | Shared mailbox support response window. |
| Slack support response | Same day when available | Best for quick unblockers, not for forcing queue jumps. |

## Task 2: Use SLA Expectations Correctly

1. Use the normal workflow first and allow the expected response window to pass before escalating.
2. Do not use Slack direct messages to bypass the review queue for pull request merges or normal publishing work.
3. If the SLA window has passed, follow up through the same workflow or shared channel with the relevant WMS ID, PR, or workshop context.
4. When the request is time-sensitive because of a live event or customer commitment, say that explicitly instead of assuming the urgency is visible.
